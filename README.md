# jp-address-normalizer

日本の住所文字列を、`postal_code`・`prefecture_code`・`city_code`・`town`・`street`・`building`
に分割するPHPライブラリです。

[jp-postal-code-db](https://github.com/kobesoft-inc/jp-postal-code-db) が配布している
SQLite3データベースを参照データとして使い、都道府県名・市区町村名・町名を正規表現の
最長一致で切り出します。

## インストール

```bash
composer require kobesoft-inc/jp-address-normalizer
```

参照データとして [jp-postal-code-db](https://github.com/kobesoft-inc/jp-postal-code-db/releases/latest) の
`jp_postal_code.db` を別途ダウンロードしてください（本パッケージには同梱していません）。

## 使い方

### 住所の正規化

```php
use JpAddressNormalizer\AddressNormalizer;

$normalizer = new AddressNormalizer('/path/to/jp_postal_code.db');

$result = $normalizer->normalize(
    '北海道札幌市中央区北一条西６丁目１－２アーバンネット札幌ビル２Ｆ'
);

$result->postalCode;         // '0600001'（住所に無くても逆引きで補われる）
$result->prefectureCode;     // '01'
$result->prefectureName;     // '北海道'
$result->cityCode;           // '01101'
$result->cityName;           // '札幌市中央区'
$result->town;               // '北一条西'（jp-postal-code-db上の正式表記）
$result->townRaw;            // '北一条西'（入力側の生の表記。表記ゆれがあれば元のまま）
$result->aza;                // ''（DBに無い小字名が番地の前に残った場合はここに入る）
$result->street->raw;        // '６丁目１－２'
$result->street->chome;      // 6
$result->street->banchi;     // 1
$result->street->banchiSub;  // 2
$result->street->go;         // null
$result->street->format();   // '6丁目1-2'
$result->building;           // 'アーバンネット札幌ビル２Ｆ'
$result->format();           // '北海道札幌市中央区北一条西6丁目1-2アーバンネット札幌ビル２Ｆ'
$result->raw;                // '北海道札幌市中央区北一条西６丁目１－２アーバンネット札幌ビル２Ｆ'（入力そのまま）
$result->toArray();          // 連想配列として取得
```

すでに開いている`PDO`接続を使い回したい場合は`AddressNormalizer::fromPdo(PDO $pdo)`を使ってください。

### 2つの住所の一致比較

```php
use JpAddressNormalizer\AddressMatcher;
use JpAddressNormalizer\MatchLevel;

$matcher = new AddressMatcher($normalizer);

$level = $matcher->compare(
    '北海道札幌市中央区北1条西5丁目2-1',
    '北海道札幌市中央区北一条西５丁目２−１'
);
// MatchLevel::Exact（表記揺れを正規化した上で完全一致）

$level = $matcher->compare(
    '北海道札幌市中央区北一条西5丁目2-1',
    '北海道札幌市中央区北一条西5丁目9-9'
);
// MatchLevel::Chome（丁目まで一致、番地が異なる）
```

**MatchLevel（一致レベル）**

| レベル | 意味 |
|--------|------|
| `Exact` | 完全一致（建物名まで同一） |
| `Go` | 号まで一致（建物名が異なる） |
| `Banchi` | 番地まで一致（号以降が異なる） |
| `Chome` | 丁目まで一致（番地以降が異なる） |
| `Town` | 町名まで一致（丁目以降が異なる） |
| `City` | 市区町村まで一致 |
| `Prefecture` | 都道府県まで一致 |
| `None` | 一致しない |

## 対応する住所記法パターン

### 文字正規化（前処理）

入力住所に対して以下の正規化を適用してから解析します。

| 正規化 | 例 |
|--------|-----|
| Unicode NFC正規化 | NFD入力（濁点分離）→ 結合済み文字に統一 |
| 全角/半角数字 | `５丁目` ↔ `5丁目` |
| ハイフン類16種統一 | `－` `−` `‐` `–` `—` `―` `﹣` `‒` `⁃` `‑` `─` `━` 等 → `-` |
| カタカナ長音記号 | 数字間の`ー` `ｰ` → `-`（「タワー」等のカタカナ語中は保持） |
| ケ/ヶ/ヵ統一 | `千駄ケ谷` → `千駄ヶ谷` |
| 空白除去 | `東京都 渋谷区 道玄坂` → `東京都渋谷区道玄坂` |

### 町名の表記ゆれ吸収

| パターン | 例 |
|----------|-----|
| 漢数字 ↔ 算用数字 | `北一条西` ↔ `北1条西` |
| 壱弐参（旧字数詞） | `壱丁目` → `1丁目` |
| 字/大字/区の有無 | `大字桔梗` → `桔梗`、`宇陀市榛原区下井足` → `榛原下井足` |
| 町の省略 | `桔梗` → `桔梗町`（DB上の正式名にマッチ） |
| 郡の省略 | `大河原町` → `柴田郡大河原町` |
| 町名なし地域（全域1エントリ） | `南箕輪村758-3` → 町名なしで「南箕輪村一円」に自動確定 |
| 大字名が町名を繰り返す場合の省略 | `本庄町1` → `本庄町本庄`（同一市区町村に別の「本庄町」が無い場合のみ） |
| 「〜通り」の送り仮名省略 | `平和通` → `平和通り` |
| 異体字・表記ゆれ20グループ | 下表参照 |

**異体字の同値グループ:**

| グループ | 文字 |
|----------|------|
| ケ系 | ヶ ↔ ケ ↔ が ↔ ガ |
| ノ系 | ノ ↔ 之 ↔ の |
| ツ系 | ッ ↔ ツ |
| ニ系 | ニ（カタカナ）↔ 二（漢数字） |
| ハ系 | ハ（カタカナ）↔ 八（漢数字） |
| 崎系 | 崎 ↔ 﨑 |
| 塚系 | 塚 ↔ 塚(U+FA10) |
| 舘系 | 舘 ↔ 館 |
| 竈系 | 竈 ↔ 釜 |
| 條系 | 條 ↔ 条 |
| 藪系 | 藪 ↔ 薮 |
| 渕系 | 渕 ↔ 淵 ↔ 渊 |
| 曾系 | 曾 ↔ 曽 |
| 澤系 | 澤 ↔ 沢 |
| 廣系 | 廣 ↔ 広 |
| 瀉系 | 瀉 ↔ 潟 |
| 藏系 | 藏 ↔ 蔵 |
| 應系 | 應 ↔ 応 |
| 餠系 | 餠 ↔ 餅 |
| 彌系 | 彌 ↔ 弥 |
| ヱ系 | ヱ ↔ エ（旧仮名遣い） |
| ヰ系 | ヰ ↔ イ（旧仮名遣い） |

### 京都の通り名（方角表記ゆれ）

京都市内の「通り名＋方角＋町名」形式の住所（例:「河原町通御池下る」）は、方角部分の
表記ゆれをDB側の表記（「上る」「下る」、送り仮名「がる」無し）に統一してから照合します。

| パターン | 例 |
|----------|-----|
| カタカナ「ル」↔ ひらがな「る」 | `上ル` → `上る` |
| 送り仮名「がる」の有無 | `上がる` → `上る` |
| 「通」の有無 | `河原町今出川下る` ↔ `河原町通今出川下る` |
| 通り名＋方角＋丁目＋町名の複合 | `大黒町通五条上る２丁目大黒町302番地`（通り名と町名が同名でも正しく町名側を特定） |

### 番地部分（street）の記法

| 記法 | 解釈 |
|------|------|
| `5丁目2番3号` | chome=5, banchi=2, go=3 |
| `5丁目2-3` | chome=5, banchi=2, banchiSub=3 |
| `5丁目2番地3号` | chome=5, banchi=2, go=3 |
| `5-2-3` | chome=5, banchi=2, go=3（丁目省略時） |
| `5の2の3` | 「の」をハイフンと同等に扱う |
| `２−８−１`（全角） | 半角と同様に解析 |
| `二十二丁目` | 漢数字→22 |
| `Ａ丁目` | chomeLabel='A'（アルファベット丁目） |
| 地割・線 | rawのみ保持（構造化しない） |

### 都道府県・市区町村の省略

| パターン | 動作 |
|----------|------|
| 都道府県省略 | 市区町村名が全国で一意なら推測補完（例: `帯広市...` → `北海道帯広市...`） |
| 同名市区町村 | 複数都道府県に存在する場合は推測しない（例: `府中市`） |
| 郡省略 | `大河原町3-2` → `柴田郡大河原町`にマッチ |

### 郵便番号の記法

| 記法 | 例 |
|------|-----|
| ハイフン付き | `060-0001` |
| ハイフンなし | `0600001` |
| 〒マーク付き | `〒060-0001` |
| 「郵便番号」プレフィックス | `郵便番号：060-0001` |

### 郵便番号の逆引き

住所に郵便番号が含まれていなくても、町名+番地から逆引きで補完します。

| 判別方法 | 例 |
|----------|-----|
| 丁目範囲 | `１〜１９丁目`, `１、２丁目`, `３５〜３８、４１、４２丁目` |
| 番地範囲 | `１〜１３１番地`, `６７−４〜１１３−７番地`, `１０００〜` |
| 番地の複数区間（「・」区切り） | `４丁目１〜２番・１０〜２７番`（丁目内で番地が飛び飛びに分かれる場合） |
| 番地以上/以下/以降 | `４５０番地以下`, `４５１番地以上`, `６６４番地以降` |
| 丁目+番地の複合条件 | `１、２丁目、３丁目１番〜２８２番`, `３丁目２８３番以上` |
| 丁目+番+号の複合条件 | `２丁目１番１〜２３号、２番`（続く「２番」は丁目を引き継ぐ） |
| 丁目-番-号のハイフン表記 | `１−１−８`（＝1丁目1番8号として構造化） |
| 大字の有無による判別 | `大字軽井沢` ↔ `軽井沢`（DBのdetailが「大字」の有無だけを条件にする町） |
| 階数（高層ビル） | `１階`〜`５１階`, `35F`, `地階・階層不明`, `高層棟、３階` |
| 地区名テキストマッチ | `上前浪、下前浪`（京都通り名含む） |
| 複合パターン | `１〜３丁目、白滝Ｂ・Ｃ、高見`（丁目+テキストのOR） |
| catch-all消去法 | `その他`（他の候補が明確に除外された場合のみ。建物名等の固有名詞テキストは、2文字以上であれば住所に無いことも除外の根拠にする） |

確定できなかった場合は`unresolvedReason`に理由が入ります:

| 理由 | 意味 |
|------|------|
| `town_not_found` | 町名が参照データに見つからない |
| `chome_unknown` | 丁目情報が不足 |
| `banchi_unknown` | 番地情報が不足 |
| `sub_banchi_unknown` | 枝番情報が不足で境界判定不能 |
| `district_unmatched` | 地区名がマッチしない |
| `ambiguous` | 複数候補があり絞り込めない |

## 分割の仕組み

1. **postal_code**: 先頭の郵便番号表記を正規表現で抽出。
2. **prefecture_code**: 都道府県名47件の最長一致。省略時は市区町村名から推測。
3. **city_code**: 該当都道府県内の市区町村名の最長一致。郡省略にも対応。
4. **town**: 該当市区町村内の町名の最長一致。表記ゆれ・異体字を吸収。
5. **street** / **building**: 番地部分と建物名を正規表現で分割し、番地を構造化。
6. **postal_code逆引き**: town+streetから郵便番号を特定（上記の判別ロジック）。

いずれかの段階で一致するものが見つからなければ、それ以降は分割せずそのまま残します。
存在しない住所や解析できない住所を渡してもエラーにはなりません。

## 検証結果

[jp-postal-code-db](https://github.com/kobesoft-inc/jp-postal-code-db) と
[houjin-bangou-db](https://github.com/kobesoft-inc/houjin-bangou-db) の実データに対して、
本ライブラリで解析した結果が元データと一致するかを検証しています。

**都道府県・市区町村の解決精度**

| データソース | 件数 | prefecture_code一致率 | city_code一致率 | town特定率 |
| --- | --- | --- | --- | --- |
| jp-postal-code-db（offices全件） | 22,419件 | 100% | 100% | 99.47% |
| houjin-bangou-db（有効な法人、ランダム2万件） | 20,000件 | 100% | 99.98% | 97.7% |

town特定率が100%に届かない残り（jp-postal-code-dbで約0.5%）は、主に次のような
住所文字列だけからは原理的に判定できないケースです。

- 郵便局私書箱のみの住所（実際の町名を含まない）
- 土地区画整理事業地内の「○○街区」表記（通常の町名体系に属さない）
- 廃止・変更された旧地名で、参照データベースに該当する町名が存在しない
- 町名を省略して番地のみが書かれている住所

**郵便番号の逆引き精度**

| ケース | 件数 | 正解率 | 備考 |
| --- | --- | --- | --- |
| 町名が1つの郵便番号にしか対応しない | 10,000件 | 100% | |
| 丁目範囲で分かれるケース | 167件（全件） | 100% | |
| 番地の範囲・列挙で分かれるケース | 347件（全件） | 99.4% | 残りは枝番不明で判定不能 |
| 地区名・京都の通り名等 | 1,553件（全件） | 100% | 複数マッチ時は最長一致で判定 |
| 「その他」への消去法フォールバック | 174件（全件） | 100% | 判定できない枝番境界のケースは番地の範囲・列挙側に計上 |
| 丁目+番地の複合条件 | 萩原台西等 | 100% | `３丁目２８３番以上` 等の複合パターン |
| 町名なし地域（番地で区分） | 小菅村等 | 100% | DB側で番地範囲をdetailに格納 |

**郵便番号の解決率（houjin-bangou-db全件）**

houjin-bangou-dbの法人住所は正解の郵便番号を含まないため一致率の検証はできませんが、
全579万6,215件に対して`postalCode`が解決できた割合を集計しています。

| 結果 | 件数 | 割合 |
| --- | --- | --- |
| 解決 | 5,788,302件 | 99.86% |
| `district_unmatched`（地区名不一致） | 3,818件 | 0.066% |
| `ambiguous`（絞り込み不能） | 2,936件 | 0.051% |
| `chome_unknown`（丁目不明） | 1,007件 | 0.017% |
| `banchi_unknown`（番地不明） | 152件 | 0.003% |

未解決の大半は、番地しか書かれておらず丁目・地区の判別材料が住所に含まれない場合や、
DBのdetail欄が想定する区分方法（通り名・地区名）と実際の住所表記が一致しないなど、
住所文字列だけからは原理的に判定できないケースです。
