# 住所パースアルゴリズム詳細

このドキュメントは [README.md](README.md) の利用者向け説明を補い、`AddressNormalizer::normalize()`
が内部で何をしているかをステージごとに詳細に説明する。実装は`src/AddressNormalizer.php`と
`src/Internal/`配下のクラス群に対応している。

## 全体のパイプライン

```
入力文字列
  │
  ├─ 1. 前処理（Unicode正規化、ハイフン統一、空白除去）
  ├─ 2. 郵便番号の抽出（先頭の「〒060-0001」等）
  ├─ 3. 都道府県マッチング（省略時は市区町村名から推測）
  ├─ 4. 市区町村マッチング（郡省略に対応）
  ├─ 5. 町名マッチング（TownMatcher）
  │     └─ 表記ゆれ吸収、京都の通り名パターン、大字/字、異体字 等
  ├─ 6. 番地/建物名の分割（StreetBuildingSplitter）
  ├─ 7. 番地の構造化（StreetParser）→ 丁目・番地・枝番・号
  └─ 8. 郵便番号の逆引き（PostalCodeResolver、住所に郵便番号が無い場合のみ）
        └─ 丁目/番地/号/階数/テキスト/消去法の順に判定

  → ParsedAddress（postalCode, prefectureCode, cityCode, town, street, building, ...）
```

いずれかの段階でマッチしなければ、それ以降は分割を諦めてそのまま`building`に残す。
例外は投げず、常に「解決できた範囲までの`ParsedAddress`」を返す。

---

## 1. 前処理（`AddressNormalizer::preprocess()`）

入力文字列全体に対して、後続の正規表現マッチングが安定して動くように機械的な正規化を行う。

| 処理 | 例 |
|---|---|
| Unicode NFC正規化 | 濁点分離（NFD）入力 → 結合済み文字に統一 |
| 空白除去 | `東京都 渋谷区 道玄坂` → `東京都渋谷区道玄坂` |
| ハイフン類16種の統一 | `－` `−` `‐` `–` `—` 等 → `-` |
| 数字間の長音記号 | `５－２ー３`のように数字に挟まれた`ー`/`ｰ`のみ`-`化（「タワー」等は保持） |

この段階では**まだ数字の全角/半角、漢数字/算用数字、ケ/ヶ等は統一しない**。これらは
町名マッチング（ステージ5）・番地パース（ステージ7）でそれぞれ個別に吸収する。理由は、
同じ「表記ゆれ」でも町名部分と番地部分で扱い方が異なるため（後述）。

---

## 2. 郵便番号の抽出

`POSTAL_CODE_PATTERN`（`/^\s*(?:〒|郵便番号)?\s*[:：]?\s*(\d{3})-?(\d{4})\s*/u`）で
文字列先頭の郵便番号表記を切り出す。ここで見つかれば、ステージ8の逆引きはスキップされる。

**対応例**: `060-0001` / `0600001` / `〒060-0001` / `郵便番号：060-0001`

---

## 3. 都道府県マッチング

47都道府県名の候補と前方一致するかを、長い順に試す（`matchLongestPrefix`）。

### エッジケース: 都道府県名の省略

県庁所在地等、市区町村名が全国で一意に決まる場合は都道府県を推測する
（`matchCityIgnoringPrefecture`）。

- 例: `静岡市葵区...` → 静岡県静岡市葵区と推測できる（「静岡市」は全国に1つ）
- 例: `府中市...` → **推測しない**。東京都府中市・広島県府中市の2つが存在するため、
  誤った推測をするより「都道府県が特定できない」という結果を返すことを優先する。

### エッジケース: ケ/ヶ の表記ゆれ

都道府県名自体にケ/ヶを含むものは無いが、`matchLongestPrefix`はケ/ケ/ガ/ヶ/ガ系の
表記ゆれを吸収する共通ヘルパー（`normalizeKe`）を使っており、後続の市区町村マッチング
（同じ関数を使う）で「鎌ケ谷市」「四街道市」のような市区町村名の表記ゆれにも効く。

---

## 4. 市区町村マッチング

都道府県内の市区町村名候補と前方一致で照合する。

### エッジケース: 郡の省略

`大河原町3-2`のように「〇〇郡」を省略した表記に対応する（`matchCityWithGunOmitted`）。
郡名を含む正式名称（例:「柴田郡大河原町」）から郡部分を取り除いた候補
（「大河原町」）を作り、それでも前方一致を試みる。

- **衝突時は使わない**: 同じ省略形が県内の複数の市区町村にマッチしてしまう場合
  （例: 同じ都道府県に「A郡X町」と「B郡X町」が両方存在するような場合）は、
  安全側に倒してこの省略形を候補から外す。誤った市区町村を選ぶよりは
  「特定できない」ままにする。

---

## 5. 町名マッチング（`TownMatcher`）

町名マッチングは、実運用上もっとも例外パターンが多い部分。市区町村ごとに
「表記バリエーション → DB正式町名」の対応表（`candidatesForCity`）を作り、
最長一致するバリエーションを採用する。

### 5.1 バリエーション生成

DBの町名1件につき、以下の掛け合わせでバリエーションを生成する。

| 軸 | 内容 | 具体例 |
|---|---|---|
| 数字表記 | 元表記 / 漢数字→全角算用数字 / 全角算用数字→半角 / 算用数字→漢数字 | `北一条西` ↔ `北１条西` ↔ `北1条西` |
| 異体字（21グループ） | ケ/ヶ/が/ガ、ノ/之/の、崎/﨑、澤/沢 等 | `千駄ケ谷` ↔ `千駄ヶ谷` |
| 字/大字の付与 | 各バリエーションに「字」「大字」を前置 | `桔梗` → `字桔梗` `大字桔梗` |
| 字/大字の挿入 | 「町」「区」の直後に「大字」「字」を挿入 | DB`武雄町昭和` → 入力`武雄町大字昭和` |
| 町の省略 | 末尾が「町」（3文字以上）なら省いた形も候補に | DB`桔梗町` → 入力`桔梗`でもマッチ |
| 「〜通り」の省略 | 末尾「通り」の送り仮名を省いた形 | DB`平和通り` → 入力`平和通`でもマッチ |
| 大字名の繰り返し省略 | 「Ｘ町Ｘ」型の町名で、大字部分の省略形も候補に | DB`本庄町本庄` → 入力`本庄町`でもマッチ<br>※同一市区町村に独立した`本庄町`が無い場合のみ |

これらは掛け合わせで生成されるため、例えば `北一条西` は
`北一条西` `北１条西` `北1条西` に加えて、字/大字付き・挿入形も含めて
十数パターンの候補になる。**候補は文字数の長い順にソートしてから最長一致を試す**
ため、短い候補が誤って先にマッチすることはない。

#### なぜ数字は3種類（元表記・全角算用数字・半角算用数字）を用意するのか

漢数字↔算用数字の変換関数（`NumeralConverter`）は全角算用数字を正準形として
扱う。しかし実際の入力は半角算用数字（`1`）で書かれることが多いため、
半角バリエーションも別途生成しないと`北1条西`のような入力にマッチしない。

- **修正前の実際のバグ**: `北一条西`はマッチするのに`北1条西`（半角）はマッチしない、
  という状態が長く存在していた。`arabicToKanji()`・`kanjiToArabic()`が全角数字
  （`[０-９]`）しか認識しない実装だったため。`NumeralConverter::toHalfwidthDigits()`
  を追加し、全角算用数字バリエーションから半角版も生成することで解決した。

#### エッジケース: 数字だけの町名による誤マッチ（重大）

「十二」のように**町名全体が漢数字1つだけ**のケースが実在する（例: 新潟市北区）。
この町名の算用数字バリエーションは`12`（半角）になるが、これは**無関係な番地の
先頭2桁と偶然一致しうる**。

```
新潟県新潟市北区123番地
  → 誤: town="十二", street="3番地"（"123"の先頭"12"を町名として誤消費）
  → 正: town="", street="123番地"
```

**対応策**: `matchFirst()`で、マッチした変種が「数字のみ」で構成される場合に限り、
マッチ直後の文字がさらに数字（全角/半角）であれば、そのマッチを**採用しない**
ようにした（`TownMatcher::DIGIT_PATTERN`によるガード）。人間が読んでも
「十二」の続きなのか番地の続きなのか区別できない文字列なので、安全側に倒している。

副次的に見つかった実装上の注意点として、**PHPは`"12"`のような数字だけの文字列を
配列キーにすると自動的にint型にキャストする**。バリエーション候補の配列
（`variant => 正規町名`）のキーがこのケースでint化され、`str_starts_with()`に
渡すと型エラーで例外が発生していた。`matchFirst()`内で`(string) $variant`に
明示キャストして回避している。

### 5.2 京都の通り名パターン（`findAfterDirection` / `KYOTO_DIRECTION_PATTERN`）

京都市内の住所は「通り名＋方角＋（丁目）＋正式町名＋番地」という構造を取ることがある。
方角キーワード（`上ル` `上る` `上がる` `下ル` `下る` `下がる` `東入` `西入` `南入` `北入`）を
検出したら、**その直後から**改めて町名マッチを試みる。

```
京都府京都市中京区河原町通御池下る大黒町５番地
  通り名: 河原町通御池           方角: 下る
  → 方角の直後「大黒町５番地」から町名マッチ → town="大黒町"
```

#### エッジケース: 方角表記ゆれの吸収

DB側のdetail欄（後述ステージ8）は「上る」「下る」（ひらがな、「がる」無し）に
表記が統一されている。一方、実際の住所では「上ル」（カタカナ）や「上がる」
（送り仮名付き）も広く使われる。町名マッチの時点ではこれらは`KYOTO_DIRECTION_PATTERN`
の正規表現で吸収できるが、**マッチ後に切り出した通り名テキスト
（`kyotoStreet`）自体の文字列比較**（ステージ8の郵便番号逆引きでdetail欄と
突き合わせる際）でも表記を揃える必要がある。これは`AddressNormalizer::
normalizeKyotoDirection()`で「上ル/上がる→上る」「下ル/下がる→下る」
「東入ル→東入」に正規化してから逆引きに渡している。

- `京都府京都市下京区堀川通高辻上ル吉水町３４４` → `上ル`を`上る`に正規化
  → detail欄の`西堀川通高辻上る`等と文字列一致できる

#### エッジケース: 「通」の有無

実際の住所では「河原町通今出川下る」を「河原町**今出川**下る」（通を省略）と
書くこともある。ステージ8のテキストマッチ（`ParsedDetail::longestMatchLength`）
では、キーワードに「通」が含まれる場合、双方から「通」を取り除いた文字列
同士でも突き合わせるフォールバックを持つ。

```
DB detail: 河原町通今出川下る
入力:      河原町今出川下る（「通」省略）
  → 「通」を除去した「河原町今出川下る」同士で一致 → マッチ成立
```

#### エッジケース: 通り名と町名が同じ文字列を含む

「大黒町通五条上る２丁目大黒町302番地」のように、**通り名にも実際の町名と
同じ文字列（「大黒町」）が含まれる**ケースがある。素朴に先頭一致だけを試みると、
方角キーワードより手前の「大黒町」（＝通り名の一部）を町名として誤って
消費してしまう。

```
誤: town="大黒町"（通り名の先頭を誤認識）、street="", building="通五条上る２丁目大黒町３０２番地"
正: town="大黒町"（方角の後ろの本物）、street="３０２番地"
```

京都の通り名対応ロジックは方角キーワードの**後ろ**から町名マッチを試みるため
本来はこの問題を回避できるはずだが、方角キーワードの直後が「２丁目」のような
丁目表記の場合、それも読み飛ばしてから町名マッチを試みる追加のフォールバックが
`TownMatcher::match()`に入っている（`afterChome`の処理）。

### 5.3 大字/字/区の吸収（`matchWithAzaStripped` / `AZA_TOKENS_PATTERN`）

「大字」「字」はバリエーション生成時に前置候補としても扱われるが、それとは別に、
**入力側に余分な「大字」「字」「区」がある場合**は、それらを全て取り除いてから
再マッチを試みるフォールバックがある。

- `福知山市字天田` → DB上は`天田`（字を除去してマッチ）
- `宇陀市榛原区下井足` → DB上は`榛原下井足`（区を除去してマッチ）。
  平成の大合併で編入された旧町村名が「〜区」付きで書かれる一方、DB側の町名には
  「区」が含まれないことがあるため対象に追加した。

#### エッジケース: 大字の有無自体が郵便番号の判別条件になる

軽井沢町の「軽井沢」という町は、DB上「大字軽井沢」と書かれているか否か**だけ**で
郵便番号が2つに分かれる特殊なケース。

```
軽井沢町「軽井沢」の郵便番号
  389-0102: detail="大字"      ← 住所に「大字」の文字がある場合
  389-0103: detail="その他"    ← それ以外
```

町名マッチでは「大字」は前置バリエーションとして消費されてしまうため、
そのままではステージ8で「大字」の有無を判定する材料が失われる。
`AddressNormalizer::extractAzaPrefix()`で町名マッチが消費した先頭部分に
「大字」「字」が含まれていたかを記録し、逆引き用テキスト（`textForResolve`）に
含めることで、消費後も判定材料として残るようにしている。

```
大字軽井沢813番地4 → azaPrefix="大字" → textForResolve="大字813番地4"
                                        → detail="大字"とテキスト一致 → 389-0102
軽井沢池ノ頭1013番地38 → azaPrefix=null → textForResolve="池ノ頭1013番地38"
                                        → detail="大字"は不一致 → 389-0103（消去法）
```

### 5.4 町名なし地域（`matchIchien`）

「南箕輪村一円」のように、村全体が1つの郵便番号エリアで、町名の区分が
存在しない市区町村がある。この場合、住所側には町名に相当する文字列が
**一切現れない**（例: `南箕輪村758-3`）。

候補が「〜一円」で終わる1件だけの場合に限り、消費文字数0でこのエントリを
即座に確定させる（`matchIchien`）。「1件だけ」に限定しているのは、複数の
町名候補がある市区町村で誤って全域確定してしまうことを防ぐため。

### 5.5 未対応（意図的に対応していない）ケース

- **通称・愛称**: 「天神橋筋」（大阪、DB正式表記は「天神橋」）のような、
  地元で定着した愛称が使われるケース。DBに候補が存在しないため原理的に
  マッチしない。
- **通り名情報の完全な欠落**: 京都で「下丸屋町416番地」のように通り名を
  一切書かない場合、同名の町が複数の通り沿いに分かれているとどちらか
  判定できない（詳細はステージ8の`district_unmatched`を参照）。

## 6. 番地/建物名の分割（`StreetBuildingSplitter`）

町名より後ろの文字列を、番地部分（street）と建物名等（building）に分割する。
「数字・丁目・番地・番・号・線・区・の・ハイフン類」が連続する限りをstreetとみなし、
それ以外の文字が現れた時点で以降をbuildingとする（`STREET_PATTERN`）。

```
６丁目１－２アーバンネット札幌ビル２Ｆ
  → street="６丁目１-２"、building="アーバンネット札幌ビル２Ｆ"
```

### エッジケース: 「地」「目」を単独の文字クラスに含めない理由

`STREET_PATTERN`は「丁目」「地割」をキーワード単位として認識するが、「地」や
「目」を単独では認識しない。もし単独で認識すると、建物名・地名の先頭
（「地下鉄前ビル」「目黒マンション」等）を誤ってstreet側に取り込んでしまう。

### エッジケース: アルファベット丁目

「Ａ丁目」のように区画整理の経緯でアルファベットが丁目番号の代わりに
使われる地域が実在する。アルファベット単体はキーワードとして認識せず
（「ABCビル」のような建物名の先頭を誤認識するため）、**「丁目」に直接続く
場合のみ**認識する。

### エッジケース: 町名マッチで取りこぼした小字の後の番地抽出

町名マッチが「大森」で確定した後、`字北内町41番地の5`のように**小字名**
（DBの町名候補に無い、より細かい字）が番地の前に残ることがある。
`STREET_PATTERN`は先頭が「字」から始まる時点でマッチせず、以前は
文字列全体を`building`に押し込めてしまっていた（番地が一切構造化されない）。

```
修正前: 大森 字北内町41番地の5 → street="", building="字北内町41番地の5"（番地不明）
```

**対応策**: テキストが「大字」「字」で始まる場合に限り、`ANYWHERE_STREET_PATTERN`
（先頭アンカー無し）で文字列中のどこかにある番地パターンを探し、番地部分を
streetとして切り出す。「大字」「字」で始まる場合に限定しているのは、それ以外の
任意の建物名文字列に対してこの緩い検索を行うと、建物名中の無関係な数字を
番地と誤認識するリスクが高くなるため。

切り出した小字名は`building`（建物名等）ではなく、専用の`aza`フィールドに
保持する。`building`に混ぜると`format()`で「大森41-5字北内町」のように
番地の後に地名が来る、あべこべな並びで復元されてしまうため
（`aza`は町名の直後・番地の直前に位置する情報として`format()`に組み込まれる）。

```
修正後: 大森 字北内町41番地の5 → aza="字北内町", street="41番地の5", building=""
        → format() = "...大森字北内町41-5"（自然な語順で復元される）
```

- **未対応**: 「今泉」のように「字」マーカーが無い、裸の小地名+番地
  （例: `大字片丘今泉9828番地15`）。マーカーが無いため、建物名との
  区別がつかず安全に一般化できない。

### エッジケース: 字名と建物名の間に区切りが無い場合（`AddressExceptions`）

「小笠原村父島字小曲コーポパッション103」のように、字名と建物名の間に数字も
区切り文字も無いケースがある。上記の番地探索ロジックは番地（数字）が現れるまでを
字名とみなすため、このケースでは「字小曲コーポパッション」を丸ごと字名として
扱ってしまい、実在の建物名「コーポパッション」まで`aza`に混入する。

```
誤: aza="字小曲コーポパッション", street="103", building=""
正: aza="字小曲", street="103", building="コーポパッション"
```

これは実在する字名を知らなければ原理的に区別できない（区切りが無い以上、
一般的な文字パターンでは判定不能）。そのため、`AddressExceptions`クラスに
`city_code`ごとの既知の字名テーブルを持たせ、`StreetBuildingSplitter`が
最優先でこのテーブルと照合するようにした。houjin-bangou-db全件（579万件）を
検証して見つかった、現時点で判明している具体的なケースのみを収録している。

**京都の通り名が無いケースをこのテーブルに含めない理由**: 同じ仕組みで
「下丸屋町416番地」のような通り名欠落ケースに、どちらかの郵便番号を決め打ちで
登録することもできなくはない。しかしこれは「実在するデータと照合して判定する」
のではなく「わからないので推測する」ことになり、外れれば誤った郵便番号を返す。
字名テーブルとは性質が異なるため、意図的に含めていない
（詳細は`AddressExceptions`クラスのdocblockを参照）。

## 7. 番地の構造化（`StreetParser`）

ステージ6で切り出したstreet文字列を、丁目（chome）・番地（banchi）・
枝番（banchiSub）・号（go）に分解する。

### 7.1 基本パターン

| 入力 | chome | banchi | banchiSub | go |
|---|---|---|---|---|
| `5丁目2番3号` | 5 | 2 | - | 3 |
| `5丁目2-3` | 5 | 2 | 3 | - |
| `5丁目2番地3号` | 5 | 2 | - | 3 |
| `2-8-1`（丁目キーワード無し） | 2 | 8 | - | 1 |
| `5の2の3`（「の」はハイフン同等） | - | 5 | 2 | 3 |

### エッジケース: 丁目キーワード無しの3連数字

「西新宿２－８－１」のように「丁目」の文字が無くハイフンだけで区切られる
慣用表記がある。文字列が数字とハイフンのみで構成され、かつ数字が
ちょうど3つ取れる場合に限り「丁目-番-号」と解釈する
（`StreetParser`内の`isPureNumericHyphen`判定）。これを見落とすと
3つ目の数字（号）が構造化されずに捨てられてしまう。

- 数字が2つしか無い場合（`2-8`）は「丁目-番」ではなく「番-枝番」と解釈する
  （丁目キーワードが無い場合に丁目だけを推測するのは根拠が弱いため）。

### エッジケース: 漢数字の丁目・番地

「三十八丁目」のような漢数字表記も、パース前に`NumeralConverter::
kanjiToArabic()`で算用数字に変換してから処理する。「三八」のように「十」を
欠いた表記（本来は誤記だが実データに存在する）も、桁を並べた算用数字
（38）として解釈する。

### エッジケース: 地割・線

北海道の「地割」（開拓地番）や「線」（碁盤目状の区画）は、数字の抽出を
行わずrawのみ保持する。これらは丁目・番地の体系と異なるため、無理に
構造化しようとすると誤った数値を抽出してしまうリスクがあるため。

## 8. 郵便番号の逆引き（`PostalCodeResolver`）

住所に郵便番号が含まれない場合、町名+番地から郵便番号を特定する。
ある町名が1つの郵便番号にしか対応しない場合（大半のケース）はそれを
そのまま返す。複数の郵便番号に分かれる場合は、DBのdetail欄
（`jp-postal-code-db`が持つ「町名内のどの区画がどの郵便番号か」を表す
自由記述フィールド）を`ParsedDetail`でパースし、パターンに応じて判定する。

### 8.1 detailのパターン一覧（`DetailPattern`）

| パターン | detail例 | 判定材料 |
|---|---|---|
| `ChomeRange` | `１〜１９丁目`、`１、２丁目` | 丁目番号 |
| `ChomeExistence` | `丁目`（丁目の有無のみが条件） | 丁目の有無 |
| `BanchiRange` | `１〜１３１番地`、`６７−４〜１１３−７番地` | 番地（枝番含む） |
| `BanchiBound` | `４５０番地以下`、`４５１番地以上` | 番地の大小 |
| `ChomeBanchi` | `３丁目１番〜２８２番`、`４丁目３〜９番`、`４丁目１〜２番・１０〜２７番` | 丁目+番地（複数区間可） |
| `ChomeBanchiGo` | `２丁目１番１〜２３号`、`１−１−８`（ハイフン表記） | 丁目+番+号 |
| `Floor` | `１階`〜`５１階`、`地階・階層不明` | 建物の階数 |
| `Text` | `上前浪`、`下丸屋町通五条上る` | 地区名・通り名のテキスト一致 |
| `CatchAll` | `その他`、空文字列 | 他の全条件に該当しない受け皿 |

判定は上表の順（丁目+番地の複合 → 丁目 → 番地 → 階数 → テキスト → 消去法）で
試し、最初に一意に決まった時点で確定する。

### 8.2 複合パターンのエッジケース

#### 丁目+番+号の3階層（`ChomeBanchiGo`）

「２丁目１番１〜２３号、２番」のように、丁目→番→号の3階層で郵便番号が
分かれる地域がある（東京都新宿区西早稲田等）。「、２番」の部分は丁目を
省略した続きの表記で、直前の`ChomeBanchiGo`の丁目を引き継いで
「2丁目2番（号は問わない）」を意味する（`ParsedDetail::parseItems()`の
`$carryChomeRanges`）。

```
西早稲田２丁目１番２３号   → 1番1〜23号の範囲内 → 162-0051
西早稲田２丁目１番２４号   → 範囲外 → その他(169-0051)
西早稲田２丁目２番１号     → 「2番」に一致 → 162-0051
西早稲田３丁目１番１号     → 丁目が対象外 → その他(169-0051)
```

また「１−１−８」のようなハイフン区切りの表記（名古屋市中村区「名駅」等）も
同じ`ChomeBanchiGo`として構造化し、「1丁目1番8号」と等価に扱う。

#### 番地の複数区間（`ChomeBanchi`の`banchi`が配列）

「４丁目１〜２番・１０〜２７番」のように、同じ丁目内で番地が飛び飛びの
複数区間に分かれることがある（東京都江戸川区西瑞江等）。「・」区切りの
各区間を独立した範囲として保持し、いずれかに該当すればマッチとする。

#### 丁目+番地条件と丁目のみ条件の混同を避ける

`ChomeBanchi`（丁目+番地の複合条件）を持つdetailに対して、丁目だけを見る
判定（`matchesPureChome`）や番地だけを見る判定（`evaluateBanchi`）を
誤って適用すると、番地条件を見落として過剰にマッチしてしまう。

```
修正前の誤り: 「1〜3丁目、4丁目1〜14番」というdetailに対し、
              chome=4, banchi=20（範囲外）の住所が「丁目一致」だけで
              誤ってこのdetailにマッチしてしまっていた
修正後:       ChomeBanchi/ChomeBanchiGoを含むdetailは、丁目のみ・番地のみの
              判定経路から明示的に除外し（`matchesPureChome`/
              `discriminatesByPureBanchi`）、複合条件専用の評価関数
              （`evaluateChomeBanchi`/`evaluateChomeBanchiGo`）でのみ判定する
```

### 8.3 消去法フォールバック（catch-all）

「その他」（またはdetail空文字列）は、他のどの条件にも当てはまらない住所の
受け皿として使われる。これを安全に採用できるのは、**他の候補全てが
「明確に該当しない」と判定できる場合に限る**（`allDefinitelyExcluded`）。

| 他候補の種類 | 除外できる条件 |
|---|---|
| `ChomeBanchi`/`ChomeBanchiGo` | 丁目・番地の両方が判明していて、条件に一致しないと判定できる場合 |
| `ChomeRange`（`ChomeExistence`含む） | 丁目が判明していて範囲外、または丁目が「無い」ことが確定している場合 |
| `BanchiRange`/`BanchiBound` | 番地が判明していて範囲外と判定できる場合 |
| `Floor` | 階数条件は消去法の対象外（保守的に「除外できない」扱い） |
| `Text`（2文字以上のキーワード） | キーワードが住所中に一切現れない場合 |
| `Text`（1文字の方角語等） | **対象外**。自由記述の省略・言い換えの可能性があるため |

#### エッジケース: テキスト条件の不在を除外根拠にできるか

「大字」「○○ビル」のような具体的な固有名詞は、住所に一切書かれていなければ
「該当しない」と確信できる。一方「南」「東」のような1文字の方角語は、
実際には該当していても自由記述の都合で省略されている可能性があり、
不在を根拠に除外するのは危険。`isConfidentlyExcludableByTextAbsence()`で
キーワードの文字数（2文字以上）により線引きしている。

```
北海道札幌市中央区北一条西5丁目9-9
  detail候補: 「南」（テキスト、1文字）/「その他」
  → 「南」の不在だけでは除外できない（自由記述の可能性）→ 判定不能のまま

長野県北佐久郡軽井沢町軽井沢池ノ頭1013番地38
  detail候補: 「大字」（テキスト、2文字）/「その他」
  → 「大字」の不在で除外できる → 「その他」に確定
```

### 8.4 未対応（原理的に判定不能）なケース

以下は住所文字列自体に判定材料が含まれておらず、アルゴリズムの改善では
解決できない。`unresolvedReason`に理由コードが入る。

| ケース | 例 | 理由コード |
|---|---|---|
| 郵便局私書箱のみ | `（成田国際空港郵便局私書箱第1001号）` | `district_unmatched` |
| 土地区画整理事業地内 | `○○街区`（通常の町名体系に属さない） | `district_unmatched` |
| 通り名情報の欠落（京都） | `下丸屋町416番地`（同名町が複数通りに分かれるが通り名が無い） | `district_unmatched` |
| 丁目が全く書かれていない | `栄町5番14号`（DBは丁目区分あり、住所は丁目省略） | `chome_unknown` |
| 番地の境界がちょうどで枝番不明 | `67-4〜113-7番地`の境界の「67番地」（枝番無し） | `sub_banchi_unknown` |
| DB自体に町名+detailが両方空の重複行 | 茨城県猿島郡境町の一部（データ側の制約） | `ambiguous` |

これらは`README.md`の「郵便番号の逆引き」節にある通り、houjin-bangou-db
全件（579万件超）での実測で0.1〜0.2%程度に収まることを確認している。

## 9. 2住所の比較（`AddressMatcher`）

`normalize()`の結果同士を比較し、どのレベルまで一致するかを`MatchLevel`で返す。
比較は都道府県→市区町村→町名→丁目→番地→枝番→号→建物名の順に行い、
**最初に不一致になった段階**のレベルを返す（表記ゆれを正規化した後の比較なので、
`AddressMatcher`自体は表記ゆれを意識する必要が無い）。

| レベル | 意味 |
|---|---|
| `Exact` | 完全一致（建物名まで同一） |
| `Go` | 号まで一致（建物名が異なる） |
| `Banchi` | 番地まで一致（号以降が異なる） |
| `Chome` | 丁目まで一致（番地以降が異なる） |
| `Town` | 町名まで一致（丁目以降が異なる） |
| `City` | 市区町村まで一致 |
| `Prefecture` | 都道府県まで一致 |
| `None` | 一致しない、または片方が解決不能 |

片方の`chome`/`banchi`等が`null`（構造化できなかった）の場合は、
「一致していない」とはみなさず、判明している方の情報で判定を続ける
（例: 片方だけ枝番が無い場合、枝番の比較はスキップして号の比較に進む）。
